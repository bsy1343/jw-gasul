## 1. 개요

비계 시공(재원가설) 사업에서 현장에 투입되는 **외국인·한국인 근로자**의 인적사항·신분증·계좌·비자/교육 유효기간을 관리하고, 현장 투입 명부를 생성·배포하는 사내 웹 애플리케이션.

| 항목 | 내용 |
|---|---|
| 프로젝트명 | jw-gasul (재원가설) |
| GitHub 레포 | `jw-gasul` |
| 패키지 | `com.jwgasul` (group `com.jwgasul`) |
| DB 스키마 | `jwgasul` |
| 도메인 | `jw-gasul.com.dev` |
| 앱 표시명 | 재원가설 인원관리 |
| 사용자 | 사장 · 경리 등 사무 인력 1~3명 (단일 권한, 역할 분리 없음) |
| 배포 환경 | Mac Mini M4 홈서버 (OrbStack/Docker) |
| 접근 경로 | Cloudflare Tunnel + Nginx Proxy Manager |

### 핵심 업무 흐름

```
현장 등록 → 근로자 등록(외국인/한국인) → 서류·계좌 등록 → 유효기간 확인
   → 현장별 명부 생성(랜덤/선택) → 엑셀 출력
```

## 3. 데이터 모델

### 3.1 `worker` — 근로자

| 컬럼 | 타입 | 필수 | 설명 |
|---|---|---|---|
| id | BIGSERIAL PK | Y | |
| worker_type | VARCHAR(10) | Y | `FOREIGN` / `KOREAN` |
| name_ko | VARCHAR(50) | Y | 한국 이름 |
| name_foreign | VARCHAR(100) | **N** | 외국 이름 (선택) |
| birth_date | DATE | Y | 생년월일 (시간 없음) |
| phone | VARCHAR(20) | Y | 휴대폰 (숫자만 저장) |
| nationality | VARCHAR(50) | N | 국적 |
| visa_grade | VARCHAR(20) | N | 비자등급 (E-9, H-2, F-4 등) |
| visa_expire_date | DATE | Y | 비자 만료일, **기본값 `9999-12-31`** |
| edu_complete_date | DATE | N | 기초안전보건교육 이수일 |
| edu_expire_date | DATE | N | 교육 유효기간 만료일 |
| is_fixed | BOOLEAN | Y | 고정 인원 여부 (기본 false) |
| memo | TEXT | N | 비고 |
| created_at / updated_at / deleted_at | TIMESTAMP | | soft delete |

**설계 노트**

- `DATE`는 PostgreSQL에서 **날짜만** 저장한다. 시간 성분 없음. Java 매핑은 `LocalDate`
- `visa_expire_date`는 NOT NULL + `DEFAULT '9999-12-31'`
  - 한국인 또는 만료일 미상인 경우 이 값이 들어간다
  - **화면 표시 규칙**: 값이 `9999-12-31`이면 만료일·D-day를 표시하지 않고 `-` 로 렌더링
  - 만료 임박 집계(F-04)에서도 제외
- `worker_type = KOREAN` 인 경우 비자 관련 필드는 폼에서 숨김
- 재직 여부 컬럼 없음. 퇴사자는 soft delete 처리
- **유니크: `(name_ko, birth_date, phone) WHERE deleted_at IS NULL`** (부분 유니크 인덱스)
  - 비계 현장 특성상 재입사가 잦다. 전체 유니크로 두면 퇴사(soft delete)한 동일인을 재등록할 때 유니크 위반으로 막힌다
  - 부분 유니크로 두면 "삭제되지 않은 행"끼리만 중복을 막으므로, 재입사 시 **새 worker 행을 생성**한다. 과거 명부·감사 이력은 이전 `worker_id`로 그대로 보존된다
  - PostgreSQL 부분 인덱스: `CREATE UNIQUE INDEX ... ON worker (name_ko, birth_date, phone) WHERE deleted_at IS NULL`
- **교육 유효기간 자동 계산**: `edu_complete_date`(이수일)를 입력하면 `edu_expire_date = edu_complete_date + 유효기간`으로 서버가 자동 산정한다
  - 유효기간은 근로자 유형별 설정값: `app.edu.valid-years.foreign`(기본 2), `app.edu.valid-years.korean`(기본 2)
  - 자동 계산이 원칙이며 사용자가 만료일을 직접 입력하는 UI는 두지 않는다 (이수일만 입력)
  - 한국인도 교육 이수증·만료 관리를 동일하게 적용한다 (유형별 유효기간만 설정으로 분리)
- 인덱스: `worker_type`, `visa_expire_date`, `edu_expire_date`, `is_fixed`, `deleted_at`

### 3.3 `worker_account` — 계좌 (인원당 최대 3개)

| 컬럼 | 타입 | 필수 | 설명 |
|---|---|---|---|
| id | BIGSERIAL PK | Y | |
| worker_id | BIGINT FK | Y | |
| bank_name | VARCHAR(30) | Y | 은행명 |
| account_number | VARCHAR(30) | Y | 계좌번호 (하이픈 제외 저장) |
| account_holder | VARCHAR(50) | Y | 예금주 |
| is_primary | BOOLEAN | Y | 주계좌 여부 |
| sort_order | SMALLINT | Y | 표시 순서 (1~3) |
| memo | VARCHAR(100) | N | 예: "비상금", "가족 명의" |
| created_at / updated_at | TIMESTAMP | | |

**규칙**

- 인원당 최대 **3개**. 4번째 등록 시도 시 차단
- 주계좌(`is_primary = true`)는 인원당 **1개만** 허용 (부분 유니크 인덱스)
- 계좌 0개도 허용 (필수 아님)
- 예금주가 근로자 본인 이름과 다르면 등록 시 확인 경고 표시 (차단은 아님)
- **마스킹**: 목록·상세 기본 표시는 `국민 1234-**-**5678` 형태. 클릭 시 전체 표시 + 감사 로그(`VIEW`) 기록
- 은행명은 코드 테이블 없이 자유 입력하되, 자주 쓰는 은행 셀렉트 제공

**계좌번호 저장 방식**

- **DB 암호화 미적용.** 평문 저장
- 보호는 애플리케이션 레벨에서 처리
  - 화면 기본 마스킹 (`국민 1234-**-**5678`), 전체 노출 시 감사 로그 기록
  - 계좌 포함 엑셀 다운로드 시 감사 로그 기록
  - DB는 홈서버 내부에서만 접근 가능, 외부 포트 미개방
- ⚠️ 잔여 위험: DB 덤프 파일이 유출되면 계좌번호가 그대로 노출된다. **백업 파일 보관 위치의 접근 통제가 유일한 방어선**이므로, 덤프를 외부 클라우드에 올릴 경우 파일 단위 암호화(예: `gpg`, 압축 시 암호 설정)를 적용할 것
- 향후 필요 시 `AttributeConverter` 기반 AES 적용 가능 (컬럼 타입을 `VARCHAR(255)`로 확장 후 재암호화 배치)

### 3.5 `site` — 현장

| 컬럼 | 타입 | 필수 | 설명 |
|---|---|---|---|
| id | BIGSERIAL PK | Y | |
| name | VARCHAR(100) | Y | 현장명 (예: OO공장 3라인) |
| client_name | VARCHAR(100) | N | 발주처 |
| address | VARCHAR(255) | N | 주소 |
| start_date / end_date | DATE | N | 공사 기간 |
| is_active | BOOLEAN | Y | 진행 여부 (기본 true) |
| memo | TEXT | N | |
| created_at / updated_at | TIMESTAMP | | |

**설계 노트**

- 현장별 상시 투입 인원 매핑(`site_worker`)은 **두지 않는다.** 인원 교체가 잦아 매핑이 금방 낡고, 갱신 누락 시 퇴사자가 명부에 자동 포함되는 위험이 있음. 랜덤/선택 명부가 그 역할을 대신함
- 전역 고정 인원(`worker.is_fixed`)만으로 상시 인력을 관리
- 유니크: `name`
- 종료된 현장은 `is_active = false` — 명부 생성 시 드롭다운에서 기본 숨김

### 3.6 `roster` / `roster_member` — 명부 이력

| roster | id, site_id(FK, nullable), title, target_date, type(`RANDOM`/`MANUAL`), created_by, created_at |
|---|---|
| roster_member | id, roster_id, worker_id(FK), **snap_name_ko, snap_name_foreign, snap_phone, snap_birth_date** |

- 현장 선택 시 `site_id` 저장, 등록되지 않은 임시 현장은 `title` 자유 입력으로 대체
- 현장별 명부 이력 조회 가능

**명부 이력 스냅샷 (인적사항 고정, 계좌 실시간)**

- `roster_member`에 명부 **생성 당시**의 인적사항(`snap_name_ko`, `snap_name_foreign`, `snap_phone`, `snap_birth_date`)을 복사 저장한다
  - 이후 근로자 정보가 수정되거나 soft delete(재입사로 별도 행 생성 포함)되어도, 과거 명부는 **생성 시점 값 그대로** 재출력된다. 엑셀 모드 A/B 재출력이 항상 일관된다
  - `worker_id`는 함께 보관하여 "현재 어떤 인원인지" 추적은 가능하되, **표시·엑셀 출력은 스냅샷 값을 사용**한다
- **계좌(모드 C)는 스냅샷하지 않고 조회 시점의 `worker_account`를 실시간 조회**한다
  - 근거: 송금은 "지금 유효한 계좌"로 보내야 한다. 과거 계좌를 스냅샷하면 계좌 변경 후 옛 계좌로 재송금하는 위험이 있다. 또한 계좌 평문을 명부 테이블에 중복 적재하는 것을 피한다
  - 스냅샷된 근로자가 재입사로 새 `worker_id`를 갖거나 삭제된 경우, 계좌를 확인할 수 없으므로 모드 C 재출력 시 해당 행은 계좌 공란 + 경고 표시

## 4. 기능 요구사항

### F-01. 로그인

- Form 로그인, 세션 방식
- 미인증 시 `/login` 리다이렉트
- **로그인 실패 5회 → 해당 `username` 기준 일정 시간 차단** (IP 기준 아님)
  - 사용자 1~3명 소규모이므로 계정(username) 단위 잠금이 자연스럽다. IP 기준으로 잠그면 Cloudflare Tunnel 뒤에서 모든 요청이 같은 IP로 보여 한 명의 실패가 전원을 잠글 수 있다
- **클라이언트 IP 식별**: Cloudflare Tunnel 뒤에 있으므로 서버가 직접 보는 원격 주소는 터널/프록시 IP다. **`CF-Connecting-IP` 헤더(없으면 `X-Forwarded-For` 첫 항)를 신뢰**하여 실제 IP를 복원하고 `audit_log.ip_address`에 기록한다
  - 헤더 위조 방지: 애플리케이션은 **Cloudflare Tunnel을 통해 들어온 요청만** 처리한다는 전제(외부 포트 미개방, Nginx Proxy Manager 경유)에서 헤더를 신뢰한다. Spring의 `ForwardedHeaderFilter` 또는 커스텀 필터로 복원한다
- 로그인 사용자명을 감사 로그 및 `roster.created_by`에 기록

### F-02. 근로자 등록 / 수정 / 삭제

- **등록 시 유형 선택**: 외국인 / 한국인 → 선택에 따라 폼 필드 동적 표시

  | 필드 | 외국인 | 한국인 |
  |---|---|---|
  | 한국 이름 | 필수 | 필수 |
  | 외국 이름 | 선택 | 숨김 |
  | 생년월일 / 휴대폰 | 필수 | 필수 |
  | 국적 | 선택 | 숨김 |
  | 비자등급 / 비자만료일 | 표시 (미입력 시 `9999-12-31`) | 숨김 (`9999-12-31` 자동) |
  | 교육 이수일 / 만료일 | 선택 | 선택 |
  | 신분증 앞·뒤, 이수증 | 선택 | 선택 |
  | 계좌 (최대 3개) | 선택 | 선택 |

- **사진 3종 업로드**: 신분증 앞 / 신분증 뒤 / 이수증
  - **클라이언트 측 전처리**: 브라우저에서 업로드 직전 **리사이즈(장변 1600px) + EXIF 회전 보정 + JPEG 변환**을 수행한다
    - 근거: 서버(Java `ImageIO`)는 HEIC 디코딩 코덱이 없어 아이폰 기본 HEIC를 그대로 받으면 리사이즈·엑셀 삽입 단계에서 깨진다. 클라이언트에서 JPEG로 변환해 올리면 서버에 네이티브(libheif 등) 의존성을 추가하지 않아도 된다
    - 아이폰 HEIC → 브라우저 Canvas/디코더로 JPEG 변환 후 전송 (예: `heic2any` 등 라이브러리 검토, Context7로 최신 API 확인)
  - **서버는 검증만 수행**: 확장자 + MIME 화이트리스트를 **jpg/png로 제한**하고 HEIC 수신은 거부한다. 서버 측 리사이즈·EXIF 보정·HEIC 폴백 디코딩은 두지 않는다
    - 트레이드오프: 클라이언트 변환이 불가능한 구형 브라우저·JS 비활성 환경 또는 PC에서 HEIC 원본 직접 업로드는 **불가**하다. 실사용 단말이 최신 모바일·데스크톱 브라우저라는 전제 하의 결정. 검증 실패 시 "지원 형식(jpg/png)으로 변환 후 업로드" 안내
  - 각 최대 10MB (변환 후 기준)
  - 슬롯별 개별 재업로드·삭제, 미리보기 제공
- **계좌 입력**: 행 추가/삭제 방식으로 최대 3행. 주계좌 라디오 선택
- 고정 인원 체크박스
- soft delete (`deleted_at`) — 명부 이력 보존
- 검증: 휴대폰 형식, 생년월일 과거일, 계좌번호 숫자·하이픈만

### F-03. 근로자 목록

- **탭 구분**: 전체 / 외국인 / 한국인
- 컬럼: 유형 / 한국명 / 외국명 / 생년월일 / 연락처 / 비자만료 / 교육만료 / 서류 / 계좌 / 고정 / 교육듣기 / 관리
  - 한국인 행의 비자만료 칸은 `-`
  - 계좌 칸은 등록 수 뱃지(예: `2`), 없으면 회색 `0`
- 검색: 이름(한국·외국), 연락처
- 필터: 유형, 비자등급, 국적, 고정 여부, 비자 상태, 교육 상태, 서류 미비, 계좌 미등록
- 정렬: 비자만료일 오름차순 기본 (`9999-12-31`은 후순위로 밀림)
- 페이징 20건

### F-04. 유효기간 만료 관리 (비자 + 교육)

기준일 = 조회 시점 `today`.

| 상태 | 조건 | 표시 |
|---|---|---|
| 해당없음 | `visa_expire_date = 9999-12-31` | `-` (집계 제외) |
| 만료 | `expire_date < today` | 🔴 "만료" |
| 임박 | `today ≤ expire_date ≤ today+7일` | 🟠 "D-N" |
| 정상 | `expire_date > today+7일` | 회색 잔여일수 |
| 미등록 | `expire_date IS NULL` | ⚪ "미등록" (교육만 해당) |

- 목록 상단 요약 배지: 비자 만료 N명 / 교육 만료 M명
- 대시보드에 임박 인원 리스트
- 명부 생성 시 만료 인원 포함되면 경고
- **교육 만료일은 이수일 입력 시 자동 계산**된다 (`edu_complete_date + app.edu.valid-years.{foreign|korean}`, 3.1 참조). 이수일이 없으면 `edu_expire_date IS NULL` → "미등록"

### F-05. 현장 관리

- 현장 CRUD (현장명, 발주처, 주소, 공사 기간, 진행 여부, 비고)
- 목록: 진행 중 / 종료 탭 구분
- 현장 상세에서 **해당 현장으로 생성된 명부 이력** 조회
- 종료 현장은 `is_active = false` — 명부 생성 드롭다운에서 기본 제외 (체크 시 표시)
- 삭제는 명부 이력이 있으면 차단, 대신 종료 처리 유도

### F-06. 랜덤 명부 생성

- 입력: **현장 선택**(드롭다운, 미등록 현장은 제목 직접 입력), **날짜**, **필요 인원 수 N**
- 규칙:
  - `is_fixed = true` 인원은 **항상 포함**되고 **N에 포함**된다
    - 예: N=10, 고정 3명 → 고정 3 + 랜덤 7
  - 고정 인원 수 > N → 경고 후 고정 인원 전원 반환 또는 N 재입력
- 대상 필터: 전체 / 외국인만 / 한국인만
- 제외 옵션:
  - [x] 비자 만료자 제외 (기본 ON, `9999-12-31`은 만료로 보지 않음)
  - [ ] 비자 임박자(D-7) 제외
  - [ ] 교육 만료자 제외
  - [ ] 서류 미비자 제외
- **가용 인원 < N 처리**: 제외 옵션·대상 필터 적용 후 뽑을 수 있는 인원이 필요 인원 N보다 적으면, **가능한 만큼만 뽑고 경고를 표시**한다
  - 예: 필터 통과 8명 + N=10 → 8명 반환, "요청 10명 중 8명만 가능 (필터로 X명 제외됨)" 경고
  - 강제로 만료자를 채워 넣지 않는다 (의도치 않은 만료 인원 투입 방지). 사용자가 N을 줄이거나 필터를 완화해 재생성할 수 있도록 제외 사유·제외 인원 수를 함께 노출한다
  - 고정 인원 수 > N 인 별도 케이스는 위 규칙(경고 후 고정 전원 반환 또는 N 재입력)을 따른다
- 결과 화면: 개별 **교체(다시 뽑기)** / **제외**
  - 교체는 아직 선택되지 않은 잔여 풀에서 대체 인원을 뽑는다. 잔여 풀이 비면 교체 불가 안내
- 결과 → 엑셀 다운로드 + 명부 이력 저장 (생성 시점 인적사항 스냅샷, 3.6 참조)

### F-07. 수동 선택 명부

- 목록 체크박스 다중 선택
- 검색/필터/페이지 전환 시 선택 유지
- 확인 화면 → 엑셀 다운로드 + 명부 이력 저장

### F-08. 엑셀 다운로드

다운로드 시 **포함 항목** 선택.

**모드 A — 기본**

| 이름 | 연락처 | 생년월일 |
|---|---|---|

**모드 B — 사진 포함**

| 이름 | 연락처 | 생년월일 | 신분증(앞) | 신분증(뒤) | 이수증 |
|---|---|---|---|---|---|

**모드 C — 계좌 포함** (송금용)

| 이름 | 연락처 | 생년월일 | 은행 | 계좌번호 | 예금주 |
|---|---|---|---|---|---|

- 모드 C는 **주계좌만** 출력. 주계좌 미지정 시 `sort_order` 1번
  - **계좌는 명부 생성 시점이 아닌 조회(다운로드) 시점의 현재 계좌를 실시간 조회**한다 (3.6 참조). 이력에서 재출력 시에도 현재 유효 계좌를 사용
  - 스냅샷 인원이 삭제·재입사로 현재 계좌를 특정할 수 없으면 해당 행 계좌 공란 + 경고
- 모드 C 다운로드는 감사 로그에 기록 (계좌 정보 반출)
- **사진 삽입 메모리 전략 (모드 B)**
  - Apache POI XSSF는 워크북 전체를 메모리에 적재하므로, 원본(장변 1600px) 대신 **엑셀 삽입 전용 썸네일(장변 약 300px)을 별도 생성**해 삽입한다. 메모리·파일 크기를 억제하고 Mac Mini(M4) 홈서버 제약에서 안전하게 한다
  - **한 명부당 인원 상한(예: 100명)** 을 두어 초과 시 분할 다운로드를 안내한다. 상한값은 설정(`app.roster.max-members`)으로 외부화
  - 썸네일은 요청 시 생성(온디맨드)하거나 업로드 시 함께 생성해 캐시할 수 있다 — 구현 단계에서 결정
  - 사진은 셀 앵커 삽입(`XSSFDrawing` + `ClientAnchor`), 행 높이 자동 조정
- 파일명: `명부_{현장명}_{yyyyMMdd}.xlsx`

### F-12. 감사 로그

권한을 분리하지 않는 대신, **누가 언제 무엇을 바꿨는지**를 기록한다.

**기록 범위: 필수 항목만 남긴다** (미확정 1 확정). 이전 "권장" 항목은 로그량 대비 실익이 낮아 초기 범위에서 제외한다.

| 구분 | 대상 | 액션 |
|---|---|---|
| 필수 | `worker` | 등록 / 수정 / 삭제 |
| 필수 | `worker_account` | 등록 / 수정 / 삭제 / **전체 조회** |
| 필수 | `worker_document` | 업로드 / 교체 / 삭제 |
| 필수 | 계좌 포함 엑셀 다운로드 | 반출 이력 |
| 제외 | `site` 변경, 신분증 사진 조회, 명부 생성 | 초기 범위 제외 (향후 필요 시 추가) |
| 제외 | 단순 목록·상세 조회 | 로그 없음 |

- 인덱스: `(entity_type, entity_id)`, `created_at`
- 조회 화면 `/audit` — 기간·사용자·대상 필터
- 근로자 상세 하단에 해당 인원 변경 이력 표시
- 보관 1년 후 자동 정리 (배치)
- **구현: 서비스 레이어에서 명시적 diff** (Envers·`@EntityListeners` 미채택)
  - UPDATE 시 수정 전 엔티티를 조회해 새 값과 필드별 비교, **변경된 필드마다 `audit_log` 행**(`changed_field`, `old_value`, `new_value`)을 생성한다
  - 근거: (1) 계좌 **전체 조회(`VIEW`)**·계좌 포함 엑셀 **반출** 같은 비-CRUD 이벤트를 CRUD와 **동일한 경로**로 일관되게 남길 수 있다. (2) 수행자(`username`)와 실제 IP(`CF-Connecting-IP`)를 서비스 레이어에서 명확히 주입할 수 있다. (3) Envers는 별도 `_AUD` 테이블을 만들어 단일 `audit_log` 방침과 충돌하고, `@EntityListeners`는 리스너 내부에서 username·IP 주입과 `VIEW` 이벤트 기록이 까다롭다
  - 감사 대상 서비스 메서드에 공통 헬퍼(예: `AuditWriter`)를 두어 중복을 줄인다

## 6. 비기능 요구사항

- **보안**
  - HTTPS 필수 (Cloudflare Tunnel)
  - 업로드 파일은 웹 루트 외부 저장, 인증된 요청만 스트리밍
  - 확장자 + MIME 화이트리스트 검증
  - 계좌번호·신분증 사진 접근 이력 기록
  - 계좌번호는 평문 저장. 화면 마스킹 + 접근 감사 로그로 보호 (3.3 참조)
- **백업 (미확정 6 확정)**: PostgreSQL 일일 덤프 + 업로드 디렉터리 주기 백업을 **홈서버 로컬(내장/연결 디스크)에만** 보관하고 **외부 클라우드로 반출하지 않는다**
  - 근거: 계좌번호가 평문이므로 덤프 유출이 유일한 실질 위험(3.3)이다. 외부로 내보내지 않으면 그 위험의 표면을 홈서버 물리 접근으로 한정할 수 있다
  - 잔여 위험: 홈서버 물리 장애 시 백업 동시 소실 가능. 로컬 백업은 별도 디스크/볼륨에 분리 저장하여 완화한다
  - **향후 외부 반출이 필요해지면** 파일 단위 암호화(`gpg`)를 **필수 전제**로 하고 그때 재결정한다 (문서에만 조건 명시, 현재는 미구성)
- **모바일 완벽 대응 (mobile-first) — 최우선 (v1.2 격상)**: 사용자가 **주로 핸드폰으로 작업**하므로 모든 화면이 모바일에서 완벽히 동작해야 한다. "반응형 되면 좋음"이 아니라 **모바일이 1순위**.
  - 모바일 기준으로 먼저 설계(작은 화면 우선, 데스크톱은 확장). 데스크톱 전용 레이아웃 금지.
  - 넓은 표 금지 → 모바일은 **카드/스택**, 데스크톱만 표. 폼은 모바일 **1열**.
  - 입력 필드 **16px 이상**(iOS 확대 방지) + 터치 타깃 충분히 크게. 네비/탭은 좁은 화면에서 가로 스크롤.
  - 사진 업로드·명부 생성 등 조작도 모바일에서 검증. (구현 규약 상세는 `CLAUDE.md` Important Notes)
- **개인정보 (미확정 4 확정)**: 신분증·계좌는 최소 수집 원칙
  - 퇴사(soft delete)한 근로자의 원본 데이터·사진은 **무기한 보관**하고 **자동 파기는 두지 않는다**. 필요 시 관리자가 **수동으로 영구 삭제**한다
  - 근거: 재입사·정산·분쟁 대응상 과거 인원 데이터가 필요할 수 있고, 사용자가 소수(1~3명)라 수동 통제로 충분하다. 명부 이력은 스냅샷(3.6)으로 이미 별도 보존되므로 원본 삭제와 무관하게 과거 명부는 유지된다
  - 트레이드오프: 개인정보 최소 보관 원칙과는 거리가 있다. 실제 파기가 필요한 정보주체 요청 발생 시 수동 영구 삭제로 대응한다
