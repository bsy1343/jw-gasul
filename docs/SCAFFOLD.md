# PRD.md & CLAUDE.md 작성 지침서

이 문서는 새 프로젝트에서 `docs/PRD.md`와 `CLAUDE.md`를 체계적으로 작성하기 위한 지침이다.
사용자와 대화하며 정보를 수집하고, 두 문서를 순서대로 생성한다.

---

## 작성 순서

```
1. PRD.md 작성 (이 지침의 Part 1)
2. PRD.md 기반으로 프로젝트 기본 구조 생성
3. /init 또는 수동으로 CLAUDE.md 작성 (이 지침의 Part 2)
```

---

## 기본 프로젝트 규칙 (모든 프로젝트 공통)

아래 규칙은 별도 언급이 없어도 **모든 프로젝트에 기본 적용**한다.
PRD.md의 아키텍처/설정에, CLAUDE.md의 Important Notes에 반영한다.
사용자가 명시적으로 변경을 요청한 경우에만 해제한다.

### 프로젝트 구조
- 프론트엔드 별도 빌드가 있는 프로젝트는 모노레포 구성을 기본으로 한다 (frontend/, backend/, db/ 등 역할별 분리)
- SSR 프로젝트(Thymeleaf, JSP, Jinja2 등)는 단일 모듈 구조도 허용한다
- 각 모듈은 독립적으로 빌드/실행 가능해야 한다

### 코딩 컨벤션
- 모든 파일 최상단에 해당 파일의 역할을 주석으로 기술한다
- 모든 함수 바로 위에 해당 함수의 기능 설명을 주석으로 기술한다
- 주석 언어는 프로젝트의 기본 언어를 따른다 (한국어 프로젝트면 한국어)

### 데이터베이스
- 애플리케이션의 DB 연결은 읽기전용을 기본으로 한다
- 쓰기가 필요한 경우 PRD에 명시적으로 기술하고, 별도 쓰기 전용 연결을 구성한다

### 기술 스택 & 라이브러리 조회
- 기술 스택 결정, 라이브러리 도입, 아키텍처 설계 시 **Context7(MCP)을 필수로 조회**한다
- 조회 절차: `resolve-library-id`로 라이브러리 ID 확인 → `query-docs`로 최신 문서·권장 패턴 조회
- 조회 대상: 최신 버전, 권장 프로젝트 구조, 설정 방법, 베스트 프랙티스
- 조회 결과를 PRD.md의 기술 스택 섹션과 CLAUDE.md의 Architecture 섹션에 반영한다
- 외부 라이브러리 import가 포함된 코드 작성 시에도 Context7으로 해당 라이브러리의 현재 API를 확인한다

### 서비스 레이어 아키텍처

외부 시스템 연동이 있는 프로젝트에 적용한다. 외부 연동이 전혀 없는 프로젝트에는 적용하지 않는다.

#### 핵심 원칙 (언어 무관)

1. **Service Interface + Implementation 분리**
   - 외부 시스템 연동이 있는 서비스는 반드시 인터페이스(추상)와 구현체를 분리한다
   - 순수 내부 로직(DB CRUD, 유틸리티)은 분리하지 않는다

2. **프로필 기반 Stub/Default 전환 + 런타임 구현체 전환**
   - `test` 프로필: Stub 구현체 활성화 (JSON Mock 데이터 로딩)
   - `!test`(production) 프로필: Default 구현체 활성화 (실 API 호출)
   - 기본 프로필은 `test`로 설정 → 외부 서버 없이 바로 실행 가능
   - 런타임 구현체 전환이 필요한 경우(예: CI 엔진, 배포 타입) `@ConditionalOnProperty` 등 설정 기반 전환도 병행한다

3. **Mock 데이터 JSON 외부화**
   - Mock 데이터는 코드에 하드코딩하지 않고 JSON 파일로 분리한다
   - 경로는 언어/프레임워크 관례를 따른다 (아래 테이블 참조)

4. **외부 통신 격리 (Adapter 또는 도메인 패키지)**
   - 외부 시스템(REST API, CLI, SDK)과의 통신 로직은 adapter 패키지로 격리하거나, 도메인별 서비스 패키지(예: git/, ci/, cd/)에 캡슐화한다
   - production 프로필에서만 활성화한다

5. **커스텀 예외**
   - 외부 시스템별 커스텀 예외 클래스를 생성한다
   - 전역 예외 핸들러에서 통합 처리한다

#### 언어별 구현 가이드

실제 구현 시 **Context7(MCP)으로 해당 프레임워크의 최신 DI/프로필 패턴을 필수 조회**한다.
아래 테이블은 핵심 매핑만 제공하며, 구체적인 API·설정 방법은 Context7 조회 결과를 따른다.

| 개념 | Java/Spring | Python/FastAPI | Node.js/NestJS | Go |
|------|-------------|----------------|----------------|-----|
| 인터페이스 | `interface` | `ABC`/`Protocol` | TS `interface` | `interface` |
| DI | 생성자 주입 | `Depends()` | `@Inject()` | Wire/수동 |
| 프로필 전환 (Stub) | `@Profile("test")` | `Settings.ENV` | `ConfigModule` | Build tags/env |
| 구현체 전환 (런타임) | `@ConditionalOnProperty` | env 분기 | `useFactory` | Build tags/config |
| Mock 데이터 위치 | `resources/mock-data/` | `tests/fixtures/` | `src/mock-data/` | `testdata/` |
| Stub 초기화 | `@PostConstruct` | `@app.on_event("startup")` | `onModuleInit()` | `init()` |

#### 적용 판별 기준

- 외부 연동이 있는 서비스 → Interface + Impl 분리 대상
- 순수 내부 로직(DB CRUD, 유틸리티) → 분리하지 않음
- 외부 연동이 전혀 없는 프로젝트 → 이 규칙 적용하지 않음

#### PRD.md / CLAUDE.md 반영 지침

- PRD.md 3.2 레이어 아키텍처에 `service/ → interface + impl/` 구조를 기술한다
- CLAUDE.md Architecture에 프로필 전환 메커니즘과 Stub/Default 패턴을 기술한다
- CLAUDE.md Important Notes에 "기본 프로필은 test, 외부 서버 없이 실행 가능"을 명시한다

### E2E 테스트
- 웹 UI가 포함된 프로젝트는 Playwright를 E2E 테스트 도구로 사용한다
- Node.js 환경: `npx playwright init-agents --loop=claude`로 테스트 에이전트(planner, generator, healer)를 설정한다
- Python 환경: `pip install playwright && playwright install`로 브라우저를 설치한다
- 기타 환경: 해당 언어의 Playwright 바인딩을 설치한다
- 웹 UI가 없는 프로젝트(API 전용, CLI 등)에는 적용하지 않는다

# Part 1: PRD.md 작성

## 목적

PRD.md는 **프로젝트의 요구사항과 설계를 정의하는 문서**다.
Claude Code가 이 문서를 읽고 프로젝트의 목적, 범위, 기술적 제약을 이해할 수 있어야 한다.

## 정보 수집

사용자에게 아래 영역의 정보를 확인한다.
**이미 충분한 영역은 건너뛰고, 부족한 영역만 질문한다.**
**한 번에 최대 3개까지만 질문한다.**

### 필수 영역

| 영역 | 확인할 것 | 부족할 때 질문 예시 |
|------|----------|-------------------|
| **목적** | 이 프로젝트가 해결하는 문제가 무엇인가? | "이 시스템이 없으면 어떤 불편이 있나요?" |
| **범위** | 포함/제외 기능이 명확한가? | "이 기능은 이번 범위에 포함인가요, 향후 계획인가요?" |
| **대상 사용자** | 누가 쓰는가? 역할 구분이 있는가? | "사용자 역할이 구분되나요? (관리자/일반 등)" |
| **기술 스택** | 언어, 프레임워크, DB 등이 결정되었는가? | "기술 스택이 정해져 있나요, 아니면 추천이 필요한가요?" |
| **운영 환경** | 배포 환경, 네트워크, 보안 제약이 있는가? | "내부망인가요? 클라우드인가요? 특별한 보안 제약이 있나요?" |

### 선택 영역 (해당 시)

| 영역 | 확인할 것 | 해당 여부 판단 |
|------|----------|---------------|
| **외부 연동** | API, DB, 파일시스템 등 외부 시스템 | 단독 시스템이 아닌 경우 |
| **인프라 정보** | 서버 IP, 포트, 인증 방식 등 | 연동 대상이 있는 경우 |
| **데이터 모델** | 핵심 엔티티와 관계 | 데이터 중심 시스템인 경우 |
| **인증/인가** | 로그인 방식, 역할별 권한 | 멀티 유저 시스템인 경우 |
| **비기능 요구사항** | 성능, 가용성, 보안 수준 | 운영 환경이 특수한 경우 |

### 질문 원칙

- 사용자가 모르겠다고 하면 합리적 기본값을 제안하고 근거를 명시한다
- 사용자가 "알아서 해줘"라고 하면 일반적 베스트 프랙티스를 적용하되, 핵심 선택지만 확인한다
- 기술 스택이 미정이면 프로젝트 성격에 맞는 옵션을 2~3개 제시하고 선택하게 한다
- **기술 스택이 결정되면 Context7(MCP)으로 해당 기술의 최신 버전, 권장 아키텍처, 디렉토리 구조를 조회하여 PRD에 반영한다**
- 기본 프로젝트 규칙(위 섹션)에 해당하는 항목은 별도 질문 없이 적용하되, 사용자에게 적용됨을 안내한다

## 심층 인터뷰 모드

사용자가 PRD 초안을 작성한 뒤 심층 보강을 원하는 경우, 또는 처음부터 인터뷰 기반으로 PRD를 만들고 싶은 경우 아래 절차를 따른다.

### 인터뷰 절차

1. `docs/PRD.md`를 읽고 현재 내용의 완성도를 파악한다
2. `AskUserQuestionTool`을 사용하여 아래 관점에서 사용자를 심층 인터뷰한다:
   - **기술적 구현**: 아키텍처 선택의 근거, 확장성 고려, 기술적 제약과 대안
   - **UI & UX**: 사용자 흐름, 화면 구성, 인터랙션 패턴, 에러 처리 UX
   - **우려 사항**: 성능 병목, 보안 취약점, 운영 리스크, 데이터 정합성
   - **트레이드오프**: 기술 선택의 장단점, 범위 조정, 일정 vs 품질
   - **엣지 케이스**: 예외 상황, 동시성 문제, 데이터 마이그레이션, 장애 복구
3. 질문은 뻔하거나 상투적이지 않아야 하며, 이전 답변을 기반으로 깊이 파고든다
4. 내용이 충분히 완성될 때까지 인터뷰를 계속 이어간다
5. 인터뷰가 끝나면 수집된 내용을 `docs/PRD.md`에 반영하여 업데이트한다

### 인터뷰 호출 방법

```
docs/SCAFFOLD.md의 심층 인터뷰 모드에 따라 docs/PRD.md를 심층 인터뷰해줘
```

또는 PRD 작성 시 처음부터:

```
docs/SCAFFOLD.md 지침에 따라 PRD.md를 작성해줘. 심층 인터뷰 모드로 진행해줘.
```

## PRD.md 작성 원칙

- **Claude Code가 읽는 문서**: 사람뿐 아니라 Claude Code가 프로젝트를 이해하는 데 쓰인다
- **구체적으로**: "사용자 관리 기능" 대신 "사용자 CRUD + 비밀번호 초기화 + 계정 활성화/비활성화"
- **결정사항 명시**: 기술적 결정과 그 이유를 기록한다 (나중에 "왜 이렇게 했지?" 방지)
- **제외 사항도 기록**: 의도적으로 뺀 것과 이유를 명시한다 (범위 확장 방지)
- **버전 관리**: 큰 변경 시 Version과 Date를 갱신한다
- **현실적으로**: 초기에 모든 섹션을 완벽하게 채우지 않아도 된다. 개발 진행하며 보강한다

# Part 2: CLAUDE.md 작성

## 목적

CLAUDE.md는 **Claude Code가 이 프로젝트에서 코드를 작성할 때 참조하는 실무 지침**이다.
PRD.md가 "무엇을 만드는가"라면, CLAUDE.md는 "어떻게 만드는가"에 해당한다.

## 작성 시점

- **방법 1**: `/init` 명령어로 Claude Code가 프로젝트를 분석하여 자동 생성
- **방법 2**: PRD.md와 프로젝트 코드를 기반으로 이 지침에 따라 수동 작성

어느 방법이든, 아래 구조를 기준으로 보강한다.
**기본 프로젝트 규칙(이 문서 상단)의 코딩 컨벤션은 반드시 Important Notes에 포함한다.**

## CLAUDE.md 구조

```markdown
# CLAUDE.md

This file provides guidance to Claude Code when working with code in this repository.

## Project Overview
<프로젝트 한 줄 설명. 기술 스택, 핵심 기능, 아키텍처 요약>

## Build and Run Commands
<빌드, 실행, 테스트 명령어. 복사-붙여넣기 가능한 코드 블록>

## Architecture
<핵심 아키텍처 패턴. 서비스 전환, 레이어 구조, 데이터 흐름 등>

## Key Configuration
<환경 설정, 설정 파일 구조, 환경변수 패턴>

## API Endpoints / Entry Points
<API 라우팅 구조, 엔드포인트 목록, URL 패턴>

## Important Notes
<네이밍 컨벤션, 주의사항, 함정, 기술적 제약>
- 모든 파일 최상단에 해당 파일의 역할을 주석으로 기술할 것
- 모든 함수 바로 위에 해당 함수의 기능 설명을 주석으로 기술할 것
- DB 연결은 읽기전용이 기본 (쓰기 연결은 별도 구성)

## Operational Decisions
<기술 결정사항과 근거, 운영 환경 특수사항>
```

## 섹션별 작성 가이드

### Project Overview
- 기술 스택 + 핵심 기능을 2~3문장으로
- Claude Code가 "이 프로젝트가 뭐하는 건지" 즉시 파악할 수 있는 밀도로

### Build and Run Commands
- 복사-붙여넣기로 바로 실행 가능한 명령어
- 프로필별 실행 방법 (dev, test, prod)
- 테스트 실행 방법

### Architecture
- **핵심 패턴만**: 프로젝트에서 가장 중요한 아키텍처 패턴 2~3개
- **모노레포 구조**: 각 모듈(frontend/, backend/, db/)의 역할과 관계를 명시
- **코드가 따라야 할 규칙**: 새 코드 작성 시 반드시 따라야 할 구조적 규칙
- **서비스/레이어 관계**: 어떤 레이어가 어떤 레이어를 호출하는가

### Key Configuration
- 환경변수 패턴과 기본값
- 설정 파일 구조와 바인딩 방식
- DB 연결 설정 (읽기전용 기본, 쓰기 연결 구분)

### API Endpoints / Entry Points
- API 라우팅 구조 (계층이 있다면 명시)
- 주요 엔드포인트 또는 진입점 목록

### Important Notes
- **기본 프로젝트 규칙의 코딩 컨벤션** (파일 주석, 함수 주석, DB 읽기전용)
- **이 프로젝트에서만 해당되는 특수 규칙**
- 언어/프레임워크별 주의사항 (예: 패키지 네이밍, import 규칙 등)
- 자주 실수하기 쉬운 포인트

### Operational Decisions
- 기술 결정사항과 **왜 그렇게 결정했는지**
- 미완료 항목과 향후 계획

## CLAUDE.md 심층 인터뷰 모드

CLAUDE.md 작성 후 보강이 필요한 경우에도 심층 인터뷰를 활용할 수 있다.

```
docs/SCAFFOLD.md의 심층 인터뷰 모드에 따라 CLAUDE.md를 심층 인터뷰해줘
```

이 경우 아래 관점에서 인터뷰를 진행한다:
- **아키텍처 결정**: 왜 이 구조를 선택했는가, 대안은 없었는가
- **코딩 컨벤션**: 팀에서 지키는 규칙이 더 있는가
- **함정과 주의사항**: 이 프로젝트에서 자주 실수하는 부분이 있는가
- **운영 특수사항**: 배포, 모니터링, 장애 대응에서 특별한 점이 있는가

## CLAUDE.md 작성 원칙

- **PRD.md와 역할 분리**: PRD는 요구사항, CLAUDE.md는 구현 지침. 내용을 중복하지 않는다
- **코드 중심**: 추상적 설명보다 실제 클래스명, 파일 경로, 명령어를 쓴다
- **새 코드 가이드**: "이 프로젝트에서 새 기능을 추가하려면 이렇게 해라"가 핵심 목적
- **진화하는 문서**: 프로젝트 구조가 바뀌면 함께 업데이트한다
- **간결하게**: 모든 코드를 설명하지 않는다. Claude Code가 코드를 직접 읽을 수 있으므로, 코드만으로는 알 수 없는 맥락/규칙/결정사항만 기록한다

# PRD.md vs CLAUDE.md 역할 비교

| 관점 | PRD.md | CLAUDE.md |
|------|--------|-----------|
| **질문** | 무엇을 만드는가? | 어떻게 만드는가? |
| **독자** | 사람 + Claude Code | Claude Code (주) + 사람 |
| **내용** | 요구사항, 범위, 설계 | 구현 패턴, 빌드, 컨벤션 |
| **갱신 시점** | 요건 변경 시 | 구조/패턴 변경 시 |
| **위치** | `docs/PRD.md` | 프로젝트 루트 `CLAUDE.md` |
| **예시** | "사용자 CRUD 기능 필요" | "사용자 API는 3-tier 패턴, 환경별 설정으로 서비스 전환" |

## 진행 방식

### 사용자가 PRD 작성을 요청한 경우

1. Part 1의 정보 수집 절차에 따라 필수 영역을 확인한다
2. 부족한 영역만 질문한다 (한 번에 최대 3개)
3. 기술 스택 확정 시 Context7(MCP)으로 최신 버전/권장 구조를 조회하여 반영한다
4. 기본 프로젝트 규칙을 적용하고 사용자에게 안내한다
5. 정보가 충분하면 PRD.md 초안을 작성한다
6. 리뷰 → 수정 → 확정

### 사용자가 심층 인터뷰를 요청한 경우

1. 대상 파일(PRD.md 또는 CLAUDE.md)을 읽는다
2. AskUserQuestionTool을 사용하여 기술적 구현, UI & UX, 우려 사항, 트레이드오프, 엣지 케이스 등 모든 측면에서 심층 인터뷰를 진행한다
3. 질문은 뻔하거나 상투적이지 않아야 하며, 이전 답변을 기반으로 깊이 파고든다
4. 내용이 충분히 완성될 때까지 인터뷰를 계속 이어간다
5. 인터뷰가 끝나면 대상 파일에 내용을 업데이트한다

### 사용자가 CLAUDE.md 작성을 요청한 경우

1. PRD.md가 있는지 확인한다 (없으면 먼저 작성을 권유)
2. 프로젝트 코드가 있으면 코드를 분석하여 패턴을 추출한다
3. Part 2의 구조에 따라 CLAUDE.md를 작성한다
4. 기본 프로젝트 규칙의 코딩 컨벤션이 Important Notes에 포함되었는지 확인한다
5. 리뷰 → 수정 → 확정

### 사용자가 둘 다 요청한 경우

1. PRD.md를 먼저 작성한다
2. PRD.md 확정 후 프로젝트 기본 구조를 생성한다
3. 생성된 코드를 기반으로 CLAUDE.md를 작성한다
